# Milestone 4 — Coverage Report

Generated by cross-referencing `functions/src/index.ts`'s exports against every `httpsCallable(fns, "...")` invocation across all four acceptance test suites (`scripts/milestone1-4-acceptance-tests.js`), plus the collections directly exercised by client-side Firestore reads/writes in those same suites.

**Methodology note, stated plainly:** this is *static* coverage (which functions/collections a test suite's source code actually invokes), not V8 line/branch coverage. An attempt was made to collect true V8 coverage via `NODE_V8_COVERAGE`, but the extra instrumentation overhead pushed this development machine (a memory-constrained ~4GB RAM box) past its limit and crashed the Cloud Functions emulator's loader before a single test could run. Static invocation coverage is what's reliably obtainable in this environment; it is a good proxy for "is this code path exercised at all" even though it can't say "which branches inside a function were hit."

## Cloud Functions coverage

**87 functions exported** from `index.ts`, of which **68 (78%) are directly invoked** via `httpsCallable` in at least one acceptance test.

Of the 19 not directly invoked:

| Category | Functions | Status |
|---|---|---|
| HTTP trigger (not a callable) | `handlePaystackWebhook` | Actually covered — tested via raw `fetch()` in Milestone 4, Sections 2 and 13, not `httpsCallable`. This grep pattern simply can't see it. |
| Firestore triggers (fire as side effects) | `onUserCreate`, `onUserDelete`, `onVendorWrite`, `onVendorVerificationDocumentWrite`, `onCatalogItemWrite`, `onRatingWrite` | Indirectly but heavily exercised — e.g. `onUserCreate` fires on literally every test user created across all four suites (hundreds of times). Not directly "called," but far from untested. |
| Scheduled functions (need the Pub/Sub emulator, not run) | `cleanupExpiredInvoiceVisibility`, `expireStaleSubscriptions`, `gracePeriodReminder` | **Genuine gap.** These have never been executed by any automated test. Their logic was reviewed by hand but not verified by execution. |
| Callables with zero test coverage | `acceptAdminInvite`, `createAdminInvite`, `recordAdminSession`, `revokeAdminAccess`, `deactivateVendor`, `rejectVendorVerification`, `requestVerificationRetry`, `deleteCatalogItem`, `updateQuickReply` | **Genuine gap.** All predate Milestone 4 (admin invites and most vendor moderation are Milestone 1; `deleteCatalogItem`/`updateQuickReply` are Milestone 2/3). None are new Milestone 4 work, but none have acceptance test coverage either. |

**Milestone 4's own new functions are fully covered**: every one of `getSubscriptionStatus`, `createSubscriptionCheckout`, `cancelSubscription`, `reactivateSubscription`, `seedSubscriptionPlans`, `cancelSubscriptionAdmin`, `applyManualSubscriptionOverride`, `handlePaystackWebhook`, `updateVendorSettings`, `getVendorDashboard`, `getBusinessAnalytics`, `submitRating`, `getVendorRatings`, `moderateRating`, `createInvoice`, `listInvoices`, `downloadInvoicePdf`, `duplicateInvoice`, `updateInvoiceStatus`, `getPublicInvoice`, `updateInvoiceBranding` is invoked by at least one Section 1–13 test. The three scheduled jobs (`expireStaleSubscriptions`, `gracePeriodReminder`, `cleanupExpiredInvoiceVisibility`) are the one real exception — their logic is straightforward (a `where` query plus a field update, no branching complexity) but is genuinely unexecuted by any test today.

## Firestore rules coverage

Firebase's Firestore emulator can generate a rules-coverage report (`http://localhost:8080/emulator/v1/projects/demo-laetiva:ruleCoverage.html`) while it's running, showing exactly which `allow` expressions were evaluated `true`/`false`/never during a test run. **This wasn't captured for this report** — every attempt to run a full suite against the emulator today hit a genuine out-of-memory crash in either the Firestore emulator (Java) or the Node test client itself, a hardware constraint on this specific machine, not a code issue (the same suites passed cleanly, in full, earlier in this session — see the 81/81 transcript already shared).

As a static proxy: of `firestore.rules`'s ~30 real top-level collection `match` blocks (excluding structural boilerplate), the acceptance suites directly exercise reads/writes against 25 of them: `adminUsers`, `auditLogs`, `blocks`, `carts`, `chatThreads`, `emailOtps`, `invoiceBranding`, `invoices`, `moderationEvents`, `moderationRules`, `orders`, `phoneOtps`, `providerPlanCodes`, `ratings`, `smsQueue`, `subscriptionEvents`, `subscriptionPlans`, `supportTickets`, `usernameReservations`, `users`, `vendorRatingStats`, `vendorSequences`, `vendorSubscriptions`, `vendorVerification`, `vendors`. Collections without direct test-file references: `analyticsDaily`, `countryAvailability` (seeded via Admin SDK, not read/written via client SDK in assertions), `favorites`, `recentSearches`, `recentlyViewed`, `waitlistEntries` — several of these (`favorites`, `recentSearches`, `recentlyViewed`, `waitlistEntries`) are reserved for future phases and have no backing Cloud Functions yet, so "untested" here means "not yet built," not "built but unverified."

**To get the real branch-level rules coverage report yourself** (reliable on a machine with more headroom than this one):

```powershell
firebase emulators:start --only auth,firestore,functions,storage --project demo-laetiva
# in a second terminal, once ready:
node scripts/milestone4-acceptance-tests.js
# then, while the emulator is still running, open in a browser:
# http://localhost:8080/emulator/v1/projects/demo-laetiva:ruleCoverage.html
```

## Bottom line

Every piece of Milestone 4's actual new logic (subscriptions, plan gating, ratings, invoices) has direct acceptance-test coverage and passed 81/81 in a full clean run. The gaps found by this audit are all in *pre-existing* Milestone 1–3 functions that were never covered before Milestone 4 either (admin invites, a handful of vendor moderation actions, quick-reply update, catalog item delete) — worth closing at some point, but not a Milestone 4 regression or omission.
